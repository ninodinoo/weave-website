Why Weave?

AI coding tools are powerful, but they don't know you. They don't know your project's architecture, your coding conventions, or how you like to break down tasks. So you end up repeating yourself — explaining context, correcting habits, re-establishing rules every session.

Weave fixes this. It's a framework that lives inside your AI tool and gives it persistent, personalized knowledge about:

  • Who you are — your role, experience level, preferred work style
  • What you're building — your tech stack, architecture decisions, project goals
  • How you want AI to behave — your rules, naming conventions, testing preferences

Instead of configuring any of this manually, Weave generates everything through a natural conversation. And it keeps improving as your project evolves — automatically.

What makes Weave different

There are other tools that let you configure AI behavior. Here's why Weave takes a different approach:

  • No templates — Everything is generated specifically for you and your project. Two developers on the same codebase will get different workflows because they work differently.
  • Lives inside your tool — Weave is not a separate app, CLI dashboard, or SaaS. It installs as native skills and agents within your existing AI editor. You never leave your workflow.
  • Conversation-based setup — The onboarding is a ~10-exchange dialog. Weave asks questions, reads your code, and proposes a workflow. You approve, adjust, or reject.
  • Auto-evolving — Background hooks track your work patterns. When your project changes significantly (new framework, new team member, shifted focus), Weave proposes updates.
  • Multi-platform — One canonical workflow definition, automatically synced to Claude Code, OpenClaw, Cursor, and Codex in their native formats.
  • Zero lock-in — One command removes everything. No residual config, no orphaned files. Your project goes back to exactly how it was before.

Installation

Weave installs via npx — no global install needed. Run this in any project directory:

$ npx weave-ai

What happens

  1. Weave shows the banner and version
  2. It scans your project for AI tool directories (.claude/, .cursor/, skills/, .codex/)
  3. You choose which platforms to install for (comma-separated numbers)
  4. Weave writes native files for each selected platform
  5. It creates the .weave/ state directory

What gets installed per platform

Platform Skills Agents Hooks Config
Claude Code 7 skills → .claude/commands/ 4 subagents → .claude/agents/ 2 → .claude/hooks/ CLAUDE.md (created or appended)
OpenClaw 7 sub-skills → skills/weave/ SKILL.md + SOUL.md + STYLE.md 2 → hooks/ AGENTS.md
Cursor 7 commands → .cursor/commands/ weave.mdc → .cursor/rules/ hooks.json + 2 scripts Native .cursor/ structure
Codex 7 skills → .agents/skills/ AGENTS.md hooks.json + 2 scripts Native .codex/ structure

Requirements

  • Node.js 18 or higher
  • A project directory (Weave installs into the current working directory)
  • At least one supported AI tool configured in the project

After installation

Open your AI tool in the project and run /weave:onboarding to start the setup conversation. The skills are immediately available — onboarding just personalizes them.

Onboarding

The onboarding is the core of Weave. It's a natural conversation (~10 exchanges) between you and your AI tool, guided by the onboarding agent. It happens in three phases:

Phase 1: The Person

Weave learns about you as a developer:

  • Your role — Are you a frontend dev? Fullstack? Data engineer? Team lead?
  • Your experience level — Junior exploring, senior optimizing, or somewhere between?
  • Your tech stack — What languages, frameworks, and tools do you work with daily?
  • Your AI preferences — Do you want the AI to plan first or dive in? Terse responses or detailed explanations? Always write tests or only when asked?
  • Your frustrations — What do you keep repeating to your AI tool? What breaks your flow?

Phase 2: The Project

Weave doesn't just ask — it reads your code too. It analyzes:

  • package.json, go.mod, Cargo.toml — to understand your dependencies
  • Directory structure — to map your architecture
  • Existing config files — to detect frameworks, linters, CI setup

Then it asks about your vision, goals, team structure, and milestones. This combination of code analysis + your answers gives Weave a deep understanding of the project.

Phase 3: Workflow Proposal

Based on everything it learned, Weave proposes a complete workflow:

  • Which agents to create — and what each one does, specifically for your stack
  • Which teams to assemble — with defined handoff workflows
  • Which rules to enforce — coding conventions, review policies, testing requirements
  • Which custom skills to add — project-specific slash commands

You review the proposal. You can adjust anything — add agents, change rules, remove things you don't need. Once you approve, Weave generates all files and syncs them to your platform.

Re-running onboarding

You can run /weave:onboarding again at any time. It will detect existing config and ask if you want to start fresh or refine the current setup.

Agents

Agents are the workhorses of Weave. Each agent is a specialized AI persona with one clear job. Think of them as team members you're onboarding — each gets a role description, specific instructions, and defined boundaries.

Agent anatomy

Every agent definition includes:

  • Role — One sentence: "You are a code reviewer specialized in TypeScript and React."
  • Responsibility — Exactly what the agent does. Not vague ("help with code") but specific ("review PRs for type safety, hook dependencies, and naming conventions").
  • Instructions — Step-by-step approach. What to check first, what to prioritize, when to escalate.
  • Tools — Which tools/files the agent can access. A reviewer reads code but doesn't write. A tester runs commands but doesn't modify source.
  • Output format — What the agent produces when done. A structured review? A list of issues? A test report?

Built-in agents

Weave ships with 4 agents that support the framework itself:

AgentJob

Custom agents from onboarding

During onboarding, the architect agent designs project-specific agents. Examples:

  • planner — Breaks features into implementation steps before any code is written
  • executor — Implements code following plans, respecting your conventions
  • reviewer — Reviews changes for quality, bugs, naming, and architectural fit
  • tester — Writes tests matching your framework (Jest, Vitest, Go testing, etc.)
  • security — Checks for OWASP vulnerabilities, injection risks, auth issues
  • documenter — Writes docs matching your style (JSDoc, README updates, etc.)

Every agent is customized. Your reviewer knows about your specific ORM's gotchas. Your tester uses your exact assertion style. Nothing is generic boilerplate.

Adding agents later

/weave:add-agent

This starts a short conversation: What should the agent do? What tools does it need? Should it join an existing team? Then it generates the definition, saves to .weave/agents/, and syncs to your platform.

Teams

Individual agents are useful, but the real power comes from teams — groups of agents that work together in a defined sequence to handle complex tasks.

Team anatomy

  • Purpose — What workflow the team handles (e.g., "feature development", "bug investigation", "refactoring")
  • Members — Which agents participate, in what order
  • Workflow — The execution sequence: who goes first, who hands off to whom
  • Rejection handling — What happens when an agent (usually the reviewer) rejects the output. Does it loop back? To which agent?
  • Trigger — A slash command that activates the team (e.g., /weave:build-feature)
  • Success criteria — How to know the team completed its job

Example: Feature Development Team

Team: feature-dev
Trigger: /weave:build-feature
Workflow:

  1. Planner    → Breaks the feature into implementation steps
  2. Researcher → Reads relevant code areas, identifies patterns
  3. Executor   → Writes the code following the plan
  4. Reviewer   → Checks quality, conventions, edge cases
     ↳ reject? → Executor gets feedback, tries again
  5. Tester     → Writes tests, runs test suite

Success: All tests pass, reviewer approved, no unresolved issues.

Example: Bug Investigation Team

Team: bug-hunt
Trigger: /weave:investigate-bug
Workflow:

  1. Researcher → Reproduces the issue, traces the root cause
  2. Executor   → Writes the fix
  3. Tester     → Verifies the fix, checks for regressions
  4. Reviewer   → Final review

Success: Bug fixed, tests pass, no regressions introduced.

Creating teams

/weave:add-team

The team builder asks about the workflow you want to automate, suggests which agents to include (existing or new ones it can create), and generates the team config with a trigger skill.

Skills

Skills are slash commands that trigger specific actions. They're markdown files with instructions for the AI — when you call a skill, the AI reads the instructions and executes them.

Built-in skills (7 total)

/weave:onboarding Starts the onboarding conversation. Interviews you, analyzes your project, proposes a workflow.
/weave:evolve Triggers the optimizer agent to analyze your current workflow and propose improvements.
/weave:status Shows your current setup — agents, teams, rules, skills, platform sync status.
/weave:sync Syncs your .weave/ definitions to all active platforms in their native format.
/weave:add-agent Interactive conversation to create a new custom agent.
/weave:add-skill Interactive conversation to create a new custom skill.
/weave:add-team Interactive conversation to define a new agent team.

Custom skills from onboarding

During onboarding, Weave may create additional project-specific skills based on your needs. Examples:

  • /weave:deploy — Runs your deployment pipeline with pre-flight checks
  • /weave:review — Triggers your review team on the current changes
  • /weave:refactor — Starts a guided refactoring session with your conventions
  • /weave:plan-sprint — Breaks down your next sprint into agent-ready tasks

Evolve

Projects change. New frameworks get added, team members join, priorities shift. Your AI workflow should adapt too — and it does, with Weave's evolution system.

Background hooks

Two hooks run automatically during your work sessions:

  • evolve-hook — Tracks which files you edit and what patterns emerge. When it detects significant changes (new framework added, new directory structure, repeated manual tasks), it nudges you to run /weave:evolve.
  • context-guard — Monitors context window usage. When it gets close to the limit, it warns you so you can start a fresh session before losing context.

Manual evolution

/weave:evolve

This triggers the optimizer agent, which:

  1. Reads your git history since the last evolution
  2. Analyzes the current codebase for changes
  3. Compares your existing agents/rules against the current project state
  4. Looks for new patterns: repeated tasks that could be skills, new tech that needs rules, stale agents that no longer match the project
  5. Proposes specific changes with explanations for each

Every proposed change is shown to you with a reason. You approve, modify, or reject each one. Nothing ever changes silently.

What evolve might suggest

  • "You added Tailwind last week. I'd like to add CSS utility rules to your reviewer agent."
  • "Your security agent hasn't been triggered in 2 months. Should I archive it?"
  • "You've manually run the same 3 commands before every deploy. Want me to create a /weave:pre-deploy skill?"
  • "Your project grew from 12 to 47 files. The planner agent needs updated context about your new directory structure."

Multi-Platform Sync

Weave's core design principle: define once, use everywhere. Your workflow is stored in a single canonical format (.weave/), and the sync agent translates it to each platform's native file format.

Why native formats matter

Each AI tool has its own way of loading instructions, skills, and agents. Weave doesn't use compatibility shims or lowest-common-denominator approaches — it generates files that are indistinguishable from hand-written platform-native configs.

The source of truth

The .weave/ directory is the canonical definition of your workflow. It contains:

  • config.json — Your profile (role, stack, preferences) and project context
  • platforms.json — Which platforms are active
  • agents/*.md — Universal agent definitions
  • skills/*.md — Universal skill definitions
  • teams/*.json — Team configurations
  • rules/*.md — Universal rules
  • history/ — Change log for evolve tracking

Platform-native file mapping

ConceptClaude CodeOpenClawCursorCodex
Skills .claude/commands/ skills/sub-skills/ .cursor/commands/ .agents/skills/
Agents .claude/agents/ SKILL.md weave.mdc AGENTS.md
Rules CLAUDE.md SOUL.md weave.mdc AGENTS.md
Hooks .claude/hooks/ hooks/ .cursor/hooks/ .codex/hooks/

How to sync

/weave:sync

Run this after making changes to .weave/ files directly (adding agents, modifying rules, etc.). The built-in skills (/weave:add-agent, /weave:add-team, etc.) trigger sync automatically.

Adding a new platform

If you start using a new AI tool, just run npx weave-ai again. It will detect the new platform and let you add it. Your existing workflow syncs immediately.

File Structure

After onboarding, your project will contain these Weave-related files:

project/
├── .weave/                    # Source of truth (platform-agnostic)
│   ├── config.json            # User profile + project context
│   ├── platforms.json         # Active platforms list
│   ├── agents/*.md            # Universal agent definitions
│   ├── skills/*.md            # Universal skill definitions
│   ├── teams/*.json           # Team configurations
│   ├── rules/*.md             # Universal rules
│   └── history/               # Change log for evolve
│
├── .claude/                   # Claude Code native files
│   ├── commands/weave:*.md    # 7 skills as slash commands
│   ├── agents/weave-*.md      # 4 subagents
│   └── hooks/                 # evolve-hook + context-guard
│
├── .cursor/                   # Cursor native files
│   ├── commands/weave-*.md    # 7 commands
│   ├── rules/weave.mdc        # Rules in .mdc format
│   ├── hooks.json             # Hook configuration
│   └── hooks/                 # Hook scripts
│
├── .agents/skills/            # Codex native skills
├── .codex/hooks.json          # Codex hook configuration
│
├── skills/weave/              # OpenClaw native skills
├── SOUL.md                    # OpenClaw identity (if OpenClaw active)
├── STYLE.md                   # OpenClaw style (if OpenClaw active)
├── AGENTS.md                  # Shared by Codex + OpenClaw
└── CLAUDE.md                  # Claude Code instructions

What goes where

  • Edit definitions in .weave/ — that's the source of truth
  • Platform directories (.claude/, .cursor/, etc.) are generated — don't edit them directly, they'll get overwritten on next sync
  • Files like CLAUDE.md and AGENTS.md may contain non-Weave content too — Weave uses marked sections so it only modifies its own parts

Git

Everything in .weave/ and the platform directories should be committed to git. This way your entire team gets the same workflow, and it's versioned alongside your code.

Uninstall & Purge

Weave has zero lock-in. Two commands let you remove everything cleanly.

Uninstall — remove from project

$ npx weave-ai uninstall

This requires two confirmations:

  1. Type yes to confirm you want to uninstall
  2. Type UNWEAVE to finalize (prevents accidental runs)

What it removes:

  • The entire .weave/ directory
  • All weave- prefixed files in .claude/commands/, .claude/agents/, etc.
  • All weave- prefixed files in .cursor/commands/, .cursor/rules/, etc.
  • Weave skills from skills/weave/ and .agents/skills/
  • Hook scripts and hook configuration entries
  • Weave-generated SOUL.md, STYLE.md (if fully Weave-owned)
  • The Weave section from shared files (CLAUDE.md, AGENTS.md) — other content stays untouched

Purge — remove from system

$ npx weave-ai purge

This requires three confirmations (the stakes are higher):

  1. Type yes to confirm
  2. Type PURGE-WEAVE to confirm system removal
  3. Type yes one final time

Purge does everything uninstall does, plus:

  • Removes the global weave-ai npm package (if installed globally)
  • Clears the npx cache for weave-ai

After purge, it's as if Weave was never on your system.

Re-installing after uninstall

Just run npx weave-ai again. Since the .weave/ directory is gone, it starts completely fresh — new onboarding, new workflow.
